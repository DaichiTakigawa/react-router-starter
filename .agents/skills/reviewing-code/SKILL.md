# ローカル変更のコードレビュー

ローカルのgit変更を総合的にレビューし、問題点と改善提案をマークダウンレポートとして出力する。コード自体の変更は行わない読み取り専用のスキル。

## トリガー

以下の場合にこのスキルを使用する：

- ユーザーが明示的にレビューを要求（「レビューして」「コードレビューお願い」「変更を確認して」「/review」など）
- ユーザーがコミットを要求した際、現在のセッションでレビュー未実施の場合は「コミット前にレビューを実行しますか？」と提案する

## ワークフロー

### 1. レビュー対象の特定

```bash
git status --porcelain
git diff --stat
git diff --staged --stat
```

**判定ロジック：**

- ステージ済み変更がある → ステージ済みのみレビュー（`git diff --staged`）
- ステージ済みなし＋未ステージ変更あり → 未ステージ変更をレビュー（`git diff`）
- 両方ある → ステージ済みを優先レビュー。未ステージ変更は別途言及
- 変更なし → 「レビュー対象の変更がありません」と報告して終了

### 2. 自動チェックの実行

変更ファイルに対して既存のlint/型チェックツールを実行する。プロジェクト全体ではなく変更ファイルのみ対象にすることで高速に完了する。

```bash
# 変更された .ts/.tsx/.js/.jsx ファイルのみ対象
CHANGED_FILES=$(git diff --name-only [--staged] | grep -E '\.(ts|tsx|js|jsx)$' | tr '\n' ' ')

# ESLint（変更ファイルのみ）
if [ -n "$CHANGED_FILES" ]; then
  pnpm eslint --no-cache $CHANGED_FILES 2>&1 || true
fi

# TypeScript 型チェック（プロジェクト全体だが高速）
pnpm typecheck 2>&1 || true
```

自動チェックの結果はレポートの「自動チェック結果」セクションに含める。ESLintやTypeScriptが検出する問題は手動レビューで重複指摘しない。

### 3. Diff分析とコンテキスト確認

```bash
# 5行のコンテキスト付きで差分取得
git diff --unified=5 [--staged]
```

差分だけでは判断できない場合、`Read` ツールでファイル全体を確認する。特に以下の場合：

- 変更された関数の呼び出し元を把握したい
- 型定義やインターフェースの全体像を確認したい
- importの追加/削除が他のコードに影響するか確認したい

### 4. 6カテゴリのレビュー実施

差分と周辺コンテキストを分析し、以下の6カテゴリで問題を検出する。

### 5. レポート出力

出力テンプレートに従ってマークダウンレポートを会話内に出力する。

## レビューカテゴリ

### A. バグ・論理エラー

実行時に不正な動作を引き起こす、または引き起こす可能性が高い問題。

確認ポイント：

- off-by-one エラー、境界条件の見落とし
- null/undefined の未チェックアクセス
- React hooks のルール違反（条件分岐内でのhook呼び出し、依存配列の不足・過剰）
- Zustand ストアで immer ミドルウェアを使用しているのに直接mutationしていない、またはその逆
- TanStack Query のクエリキー不整合（同じデータに異なるキーを使用）
- stale closure（useEffect/useCallback内で古い値を参照）
- async/await でのエラーハンドリング漏れ（try-catchなしのawait、unhandled promise rejection）
- React Router の loader/action の戻り値型不一致

### B. セキュリティ

セキュリティ上のリスクとなる問題。

確認ポイント：

- `dangerouslySetInnerHTML` によるXSSリスク、ユーザー入力の未サニタイズ
- ソースコード内のハードコードされたシークレット（APIキー、トークン、パスワード）
- IndexedDB/OPFS に保存するデータの適切性（機密データの平文保存）
- `eval()`、`Function()` コンストラクタ、`innerHTML` の使用
- loader/action でのユーザー入力バリデーション不足
- CORS設定の問題

### C. パフォーマンス

ユーザー体験やリソース使用に悪影響を与えるパフォーマンス問題。

確認ポイント：

- 不要な再レンダリング（`useMemo`/`useCallback`/`React.memo` の欠如、propsとして渡す新規オブジェクト/配列/関数リテラル）
- バンドルサイズへの影響（ライブラリ全体のimport、tree-shaking不可なimport）
- `React.lazy`/`Suspense` によるコード分割の機会
- Remotion: レンダーループ内での重い計算（`useCurrentFrame()` のコールバック内でのO(n)以上の処理）
- Zustand: ストア全体をsubscribeしている（`useStore()` を引数なしで使用、selectorを使うべき）
- TanStack Query: `staleTime`/`gcTime` の未設定や不適切な値

### D. コード品質

保守性・可読性に影響する品質上の問題。ESLint/TypeScriptが自動検出する問題は除く。

確認ポイント：

- `any` 型の使用（`unknown` + 型ガードで代替可能な場合）
- デッドコード（到達不能コード、使われていないexport）
- 単一関数/コンポーネントの責務過多（明確に分割できる場合）
- エラーハンドリングの欠如（ネットワークリクエスト、ファイルI/O等の外部操作）
- 誤解を招くコメントや変数名
- マジックナンバー/マジックストリング（定数に抽出すべき値）

### E. プロジェクト規約（ibisCut固有）

CLAUDE.md で定義されたプロジェクト固有のコーディング規約への準拠。

確認ポイント：

- `import type` の不使用（型のみのインポートに値インポート構文を使用）
- Tailwind: セマンティックカラートークンの代わりに生のカラー値を使用（`bg-red-500` → `bg-destructive`）
- Tailwind: `space-y-*`/`space-x-*` の使用（`gap-*` を使うべき）
- Tailwind: 幅と高さが同じ場合に `w-X h-X` を使用（`size-X` を使うべき）
- HugeIcons: アイコン要素にサイズクラスを付与（親要素でサイズ制御すべき）、`data-icon` 属性の欠如
- CVA（class-variance-authority）を使わずにvariant/sizeのpropsを手動管理
- クラスマージに `cn()` を使わずテンプレートリテラルや文字列結合を使用
- 相対パス `../../` の使用（`@/*` パスエイリアスを使うべき）
- コンポーネント配置: UI プリミティブが `components/ui/` 以外に配置

### F. アクセシビリティ

アクセシビリティ上の問題。

確認ポイント：

- インタラクティブ要素（ボタン、リンク、入力フィールド）の `aria-label` や `aria-labelledby` の欠如
- キーボード操作への対応不足（`onClick` のみで `onKeyDown` なし、tabIndex の欠如）
- 画像の `alt` テキスト欠如
- Radix UI プリミティブの `asChild`/`Slot` パターンの不適切な使用

## 出力テンプレート

以下のフォーマットでレポートを出力する。指摘がないセクションは省略する。

````markdown
# コードレビュー結果

## 概要

- **対象**: [staged/unstaged] の変更 ([N]ファイル, +[A]/-[D]行)
- **ファイル**: `path/to/file1.ts`, `path/to/file2.tsx`, ...

## 自動チェック結果

- **ESLint**: [問題なし / N件のエラー、M件の警告（要約）]
- **TypeScript**: [型エラーなし / N件の型エラー（要約）]

## レビュー指摘

### 🔴 重大 (Must Fix)

#### [カテゴリ] `ファイル名:行番号`

説明文。なぜ問題なのか、どのような影響があるか。
修正案がある場合：

> ```ts
> // 修正案
> ```

### 🟡 改善推奨 (Should Fix)

#### [カテゴリ] `ファイル名:行番号`

説明文。

### 🔵 提案 (Nice to Have)

#### [カテゴリ] `ファイル名:行番号`

説明文。

## まとめ

[1-2文の総評。重大な問題がなければ「コミット可能です」、あれば「重大な問題をN件修正してからコミットしてください」と明記]
````

**カテゴリタグ:** `[バグ]` `[セキュリティ]` `[パフォーマンス]` `[品質]` `[規約]` `[a11y]`

**セクション省略ルール:**

- 指摘がゼロの重大度レベルはセクションごと省略（「なし」と書かない）
- 自動チェックが問題なしの場合も一行で記載する（ツールが正常に動作したことを示すため）
- 指摘が一切ない場合は「問題は検出されませんでした。コミット可能です。」と短く報告

---

## 大量の変更への対応

差分が500行を超える場合は、ファイル単位で逐次レビューする。優先順位：

1. **新規ファイル** — 設計上の問題を早期に検出
2. **ロジック変更のあるファイル** — バグ・セキュリティリスクが最も高い
3. **フォーマット/import変更のみのファイル** — 自動チェックに任せ、手動レビューは簡略化

---

## 他スキルとの連携

- **git-commit**: レビュー完了後にユーザーがコミットを要求した場合、git-commitスキルに引き継ぐ。レビュースキル自体はコードの変更やコミットを行わない
- **simplify**: リファクタリングの機会を発見した場合、「`/simplify` でコードの簡素化を実行できます」と案内する

## 注意事項

- このスキルは**読み取り専用**。コードの変更、ファイルの作成・削除、gitの操作（コミット、ステージング等）は行わない
- ESLintやTypeScriptが検出する問題は自動チェック結果として報告し、手動レビューでは重複指摘しない
- レビュー観点はプロジェクトの技術スタック（React 19、Remotion、Zustand、TanStack Query、Tailwind CSS v4、shadcn/ui）に特化している
- ユーザーが特定の観点のみレビューを要求した場合（「セキュリティだけ見て」等）、要求された観点に絞ってレビューする
